# ADR 0004 — PVEAPIToken static-header authentication

- **Status:** Accepted
- **Date:** 2026-06-21
- **Deciders:** Frederic Jacquet

## Context

Proxmox VE offers two API auth schemes: a ticket/CSRF login flow (username +
password → time-limited ticket, refreshed periodically) and **API tokens** (a
static `USER@REALM!TOKENID=SECRET` credential sent as a header). The exporter is a
long-running, read-only process that should hold no session state and never need a
human credential.

## Decision

Authenticate with a **static `PVEAPIToken` header** and nothing else:

```
Authorization: PVEAPIToken=USER@REALM!TOKENID=SECRET
```

- **No login round-trip, no token refresh.** The header is attached to every
  request; there is no ticket to acquire or renew, and therefore no session state.
- **Least privilege.** The token only needs the built-in **PVEAuditor** role
  granted at path `/` — read-only, cluster-wide.
- **Retry policy excludes 4xx.** The resty client retries only on `429` and `5xx`
  responses. `4xx` (especially `401`/`403`, a bad or under-privileged token) is a
  configuration error, not a transient fault — retrying would only hammer the API.
- **The secret is never logged.** The token lives only in the request header. The
  `--trace` hook scrubs the `Authorization` header before emitting any trace, and
  the config loader supports `tokenSecretFile` so the secret can be mounted rather
  than placed inline.

## Consequences

**Positive**

- Stateless: no ticket lifecycle, no refresh goroutine, no clock-skew handling.
- Minimal blast radius — a leaked token is read-only and can be revoked in the PVE
  UI without touching a user password.
- Misconfigured credentials fail fast (no retry storm on `401`/`403`).
- The secret never reaches logs or traces.

**Negative / trade-offs**

- The operator must create the token and grant PVEAuditor at `/` out of band (see
  the README setup snippet).
- A static credential must be stored securely (use `tokenSecretFile` + secrets
  mounts, not inline YAML, in production).

## Alternatives considered

- **Ticket/CSRF login flow** — rejected: needs a username/password, holds session
  state, and requires periodic refresh for a read-only scraper.
- **Retry on all errors including 4xx** — rejected; `4xx` is a config error, and
  retrying it adds load without ever succeeding.
